# UI 再設計 — 議論ベース文書

本書は、画面デザインと情報設計を抜本的に見直すにあたって、**決めるべきことを整理** したものです。結論は書きません。各論点について「問い」「なぜ検討が必要か」「選択肢とそれぞれの含意」「他論点との依存関係」を並べ、話し合いで詰めていくための足場とします。

関連: [personas.md](./personas.md) / [requirements.md](./requirements.md) / [features.md](./features.md)

---

## 前提（話し合いの出発点）

現行 UI の特徴:
- **サイドバー**で 9 カードを独立にトグル
- **色の設定**は `MyColorPanel` モーダル内（メイン画面からは切り離し）
- メインエリアは flex-wrap でカードが並ぶ
- About / Contact / Help もすべてモーダル
- 初期表示: Contrast / ColorExtend / PreviewSVG の 3 つ

再設計の駆動要因:
- [requirements.md](./requirements.md) で定義した新機能群（補正サジェスト、制約付きスライダ、生成エンジン、テーマ展開）は **「色を動かしながら検証結果をリアルタイムに見る」ワークフロー**が前提
- 現状の「カード羅列 + 色はモーダル」では、色を変えるたびにモーダルを開閉する必要があり動線が破綻する
- [personas.md](./personas.md) のペルソナごとに、欲しい画面の構造が大きく異なる

## 論点 A. 第一の顧客は誰か ✅（2026-04-17 決定）

**決定**: ペルソナは絞らない。ただし **メイン動線は Web 開発関連** をメインターゲットとする（P1 Web デザイナー / P2 フロントエンドエンジニア / P3 a11y 専門家 の交差領域）。メイン動線外のユーザー（P4 学生 / P5 ブランド / P6 DataViz）は **機能の組み替え** で取りこぼさない。

→ 詳細は [ui-design.md](./ui-design.md) 参照


### 問い
[personas.md](./personas.md) の 6 ペルソナのうち、**誰を最優先** で最適化するか。

### なぜ検討が必要か
ペルソナごとに重視する機能が対立する:
- P1 デザイナー/P5 ブランド: 提案・プレビューが中核
- P2 エンジニア: エクスポート・CLI が中核
- P3 a11y 専門家: 検証レポートが中核
- P4 学生: 学習導線が中核
- P6 DataViz: 8 色固定モデルが足かせ

「全方位最適」は現行 UI がすでに陥っている罠。

### 選択肢
| 案 | 第一顧客 | 含意 |
|---|---|---|
| A1 | P1 + P5（ビジュアル提案系） | プレビュー主役、生成エンジンが目玉。a11y は二の次 |
| A2 | P2 + P3（技術検証系） | ダッシュボード的、数値とエクスポートが主役 |
| A3 | P3 単独（a11y 専門） | WCAG レポートツールに純化 |
| A4 | P4 単独（教育） | 学習モード中心、インタラクティブチュートリアル |
| A5 | P1 + P2 + P3（折衷） | モード分岐で 3 顧客をカバー（→論点 C に依存） |

### 依存
ここが決まらないと B 以降の全論点が浮く。

### 議論のたたき台
現行実装の資産（CIEDE2000 / Entropy / Preview）を活かす観点では **A1 または A5** が有力。A3 に振ると現行機能の多くが不要になる。

## 論点 B. 主要動線（ワークフロー）の定義 🤔（方向性確定、詳細は議論中）

**方向性**: Web 開発の典型フロー（**生成 → 検証 → 調整 → 展開 → 出力**）をメイン動線として採用する。ただしフローを強制するウィザードではなく、任意ステップからの出入りを許す柔らかい動線にする。

→ 詳細は [ui-design.md](./ui-design.md) 参照


### 問い
ユーザーがアプリに来て、どの順で何をするのか。1 本の主動線を敷くか、動線を定めずに自由にするか。

### なぜ検討が必要か
現行はフリーフォーム（どのカードも並列）。新機能は段階的な操作（生成→検証→調整→展開→出力）を前提としており、動線の有無で画面構造が変わる。

### 選択肢
| 案 | 動線 | レイアウト方針 |
|---|---|---|
| B1 ウィザード型 | 生成 → 検証 → 調整 → 展開 → 書き出しの 5 ステップ | ステップバー + 1 画面 1 フェーズ |
| B2 ハブ型 | 中央に配色ボード、各機能に出入り | ダッシュボード風、各機能は専用ビュー |
| B3 エディタ型 | 常に編集可能、右側に検証を常時表示 | Figma 的・IDE 的 |
| B4 ページ型 | 機能ごとに URL を分割 | `/generate` `/audit` `/export` 等 |
| B5 フリーフォーム | 現行に近い、カード並列 | 既存踏襲 |

### トレードオフ
- **B1 ウィザード**: 学習者（P4）に優しい、熟練者（P3）にはクドい
- **B3 エディタ**: 熟練者（P1/P3）に最適、初学者は迷子
- **B2 ハブ**: バランス型、実装コスト中
- **B4 ページ**: SEO と直リンク共有に強い、状態管理が複雑
- **B5 現行踏襲**: 既存機能は温存できるが新機能が乗らない

### 依存
論点 A（第一顧客）と論点 C（モード分岐）に直結。

## 論点 C. モード分岐を設けるか ✅（2026-04-17 決定）

**決定**: **明示的なモード分岐は設けない**。代わりに **パネルの組み替え（カスタマイズ可能レイアウト）** で多様な用途に対応する。

- デフォルトレイアウトは Web 開発向けメイン動線に最適化
- ユーザーは任意のパネルを追加・削除・移動できる
- 保存されたカスタムレイアウトで「監査モード」「DataViz モード」等を自作できる
- C2〜C4 の案はいずれも ❌ 却下

→ 詳細は [ui-design.md](./ui-design.md) 参照


### 問い
全ユーザーに単一 UI を出すか、モードを切り替えられるようにするか。

### 選択肢
| 案 | 分岐 | 含意 |
|---|---|---|
| C1 単一 UI | なし | 設計がシンプル、ペルソナ間の対立を 1 画面で飲む必要 |
| C2 習熟度 | 初心者 / 熟練者 | 進歩的開示（機能が段階的に出現） |
| C3 タスク | 生成 / 監査 / 書き出し | タスク別ダッシュボード（論点 B2 と相性良い） |
| C4 ロール | デザイナー / エンジニア / a11y | ペルソナ A5 と整合、UI ラベルも変わる |
| C5 ダーク UI / ライト UI のみ | 見た目のみ | 分岐ではないが UX に影響（論点 K） |

### トレードオフ
- モード分岐は **初見での発見性** を下げる（どのモードに欲しい機能があるか分からない）
- 分岐が多いほど **開発・保守コスト** が嵩む
- C2 の「進歩的開示」はモード選択を不要にできる代替案

### 依存
論点 A（A5 を選ぶなら C4 or C3 が必要）。

## 論点 D. 配色モデルそのものを維持するか

### 問い
現行の **4 役 × 2 段階（Main/Base/Accent/Text × A/B）= 8 色固定** を続けるか。

### なぜ検討が必要か
このモデルは:
- P1/P5 のブランド提案には **語彙が合う**
- P2 の Tailwind には **数値シェード（50〜950）の方が自然**
- P6 の DataViz には **任意数パレットでないと機能しない**
- 「A/B」は用途（通常/ホバー）なのか系統（主/副）なのか曖昧

### 選択肢
| 案 | モデル |
|---|---|
| D1 | 現行維持（4 役 × A/B） |
| D2 | 4 役 + 状態色（hover/active/disabled/focus）に再編 |
| D3 | Primary/Secondary/Tertiary + シェード（Material 風） |
| D4 | ブランド色 + UI 色 + フィードバック色（success/warning/error/info）の 3 分類 |
| D5 | ユーザーが枠を自由定義（最大フレキシビリティ） |
| D6 | プリセットから選ぶ（D2/D3/D4 のどれかをテンプレートとして） |

### 依存
- この決定がエクスポート形式に直結（Tailwind `primary.500` 形式に出すなら D3 必須）
- コンポーネント名（MyColorPanel、colorSetters 等）に波及する

### 議論のたたき台
Tailwind 連携を重視するなら D3、ブランド提案を重視するなら D2 が有力。D5 は自由度が高すぎて「ガイドライン適合判定」との相性が悪い。

## 論点 E. 画面レイアウトの基本型

### 問い
主画面をどんなゾーニングで構成するか。

### 選択肢（代表 5 パターン）

**E1. エディタ型（3 ペイン）**
```
┌──────────┬────────────────────────────────┬──────────┐
│          │                                 │          │
│ カラー   │      プレビュー/ステージ        │  検証    │
│ エディタ │                                 │  パネル  │
│          │                                 │          │
└──────────┴────────────────────────────────┴──────────┘
```
- 左: 8 色編集 / 中: プレビュー or 分析 / 右: 検証結果
- 常時表示で色変更→検証を即確認できる

**E2. ダッシュボード型**
```
┌───────────────────────────────────────────┐
│                 配色ボード                │
├──────────┬──────────┬─────────────────────┤
│ 検証ｽｺｱ  │ CIEDE    │ プレビュー切替      │
├──────────┴──────────┼─────────────────────┤
│ 配色ﾊﾞﾘｴｰｼｮﾝ        │ エクスポート        │
└─────────────────────┴─────────────────────┘
```
- 中央に配色ボード、周辺にサマリカード
- Lighthouse 的で一覧性が高い

**E3. ウィザード型**
```
┌─ [生成] [検証] [調整] [展開] [書出] ─┐
│                                       │
│        選択中ステップのコンテンツ     │
│                                       │
└───────────────────────────────────────┘
```
- 1 画面 1 フェーズ、進捗バー
- P4 学習者に最適

**E4. カード並列型（現行踏襲）**
- 9 カードを自由に並べる
- 新機能は新カードを追加
- 実装コスト最小、情報密度が低い

**E5. Figma 的レイヤー型**
- 中央に大きなキャンバス（プレビュー）
- 左右 + 下にドッキング可能なパネル
- パネル自体を閉じ/引き出し可能
- 高機能だが学習コスト大

### 依存
- 論点 B（動線）との整合。E3 はウィザード専用、E1 はエディタ型前提
- 論点 J（モバイル）。E1/E2 はモバイルで破綻しやすい

## 論点 F. 色の編集 UI をどこに置くか

### 問い
色を変更する UI（現行は `MyColorPanel` モーダル）を、常時表示にするか、呼び出し式にするか。

### 選択肢
| 案 | 位置 | 特徴 |
|---|---|---|
| F1 左サイドバー常駐 | 左ペインに 8 スロット | E1 と相性◎、常にアクセス可能 |
| F2 下部ドック | 画面下に帯状 | モバイル対応しやすい |
| F3 インライン編集 | 色をクリックでその場にピッカー表示 | 直感的だが編集中の他色比較が難 |
| F4 モーダル（現行） | ボタンで開閉 | 実装シンプル、動線悪い |
| F5 フローティングパネル | ドラッグで移動可能 | 自由度高、実装複雑 |

### 議論のたたき台
新機能の中核は「動かしながら検証」であり、F4（現行）では破綻する。F1 か F3 が有力。

## 論点 G. 検証結果の露出方法

### 問い
WCAG 合否・CIEDE2000・相対輝度 などの **判定結果** をどう見せるか。

### 選択肢
| 案 | 表現 | 特徴 |
|---|---|---|
| G1 専用カード（現行） | カードを開く/閉じる | 詳細閲覧型、ながら見には不向き |
| G2 統合スコアカード | AA: 6/8、CIEDE 平均: X 等のサマリを常時表示 | 一覧性高、詳細は別表示 |
| G3 インラインバッジ | 各色スロットに `AA ○` `AAA ×` バッジ | 色を選ぶ時点で可否が見える |
| G4 オーバーレイ | プレビューに「この文字は不適合」と赤枠 | 実使用に直結、実装難 |
| G5 サイドパネル常駐 | 右ペインに常時表示、色変更で更新 | E1 と合致 |

### 議論のたたき台
G2 + G5 + G4 の組合せが理想。G1 単独は現行の弱点。

## 論点 I. 機能の粒度（カード vs 統合ビュー）

### 問い
現行の 9 カードを **粒度そのまま**で持つか、**統合/再編**するか。

### 観察
現行カードには **粒度の不揃い** がある:
- `CardContrast` = 多数のペアを評価（重量級）
- `CardRelativeLuminance` = 単一指標の羅列（軽量級）
- `CardHSV` = 単色ビジュアライズ

論理的には:
- 「単色の属性」系（HSV / 輝度 / HEX/RGB 表示）は 1 つに統合可
- 「パレット評価」系（CIEDE / Entropy / Contrast）も統合可

### 選択肢
| 案 | 粒度 |
|---|---|
| I1 現行維持 | 9 カード |
| I2 中統合（3〜4 カード） | 単色分析 / パレット評価 / プレビュー / ツール |
| I3 大統合（2 ビュー） | 「分析」「操作」の 2 画面 |
| I4 完全再編 | 機能名を捨てて動線（生成/検証/調整/展開/出力）に合わせる |

## 論点 J. モバイル/レスポンシブ対応

### 問い
モバイルブラウザをサポートするか、デスクトップ専用にするか。

### 選択肢
| 案 | 対応 |
|---|---|
| J1 完全対応 | レスポンシブ、タッチ最適化、すべての機能 |
| J2 閲覧のみ | モバイルでは生成/検証の結果を読むだけ、編集不可 |
| J3 対応しない | デスクトップ専用と明言、モバイルは警告表示 |
| J4 専用ビュー | モバイル用に簡易版 UI を別実装 |

### トレードオフ
- J1 は E1/E2 の 3 ペインと相性最悪（物理的に入らない）
- J3 は潔いが、P1 デザイナーが外出先で提案する際に困る
- J4 は実装コスト倍、保守困難

## 論点 L. 既存版からの移行戦略

### 問い
既存ユーザー（いる/いないの把握も含む）にどう移行してもらうか。

### 選択肢
| 案 | 戦略 |
|---|---|
| L1 全面切替 | feature/v2 をそのまま main に、旧版は破棄 |
| L2 並存 | 旧版を `/legacy` 等で残し、新版をトップに |
| L3 段階移行 | 旧 UI 内に「新版プレビュー」リンクを設置、期間を置いて切替 |
| L4 フィーチャーフラグ | 同一コードベースで切替可能 |

### 依存
- そもそも現行 v2 ブランチが main に乗っていない（6 コミット先行中）
- 運用統計（利用者数）次第で判断変わる

## 論点 M. 技術的制約

### 問い
フレームワーク・ライブラリ・技術方針を変えるか。

### 観点
- **Next.js App Router 維持** — 静的ホスティングで足りるなら SPA 化も選択肢
- **Zustand 継続 or Context API / Jotai / TanStack Store** — エディタ型になるなら状態の粒度が変わる
- **Tailwind v3 → v4** — CSS-first API への移行
- **CSS ライブラリ導入** — shadcn/ui, Radix, Mantine, Park UI 等、UI フレームワーク採用するか
- **アニメーション** — framer-motion 継続 / View Transitions API / CSS-only に置換
- **色計算** — chroma-js 継続 / culori / colorjs.io に移行（OKLCH 等の新しい空間を扱うなら後者）

### 議論のたたき台
- OKLCH 対応が必要なら `culori` or `colorjs.io` を検討
- UI フレームワーク導入は開発速度を上げるが、独自デザインと干渉する

## 次に話し合いで決めるべき順序（提案）

1. **論点 A** を決める（30 分〜1 時間で固めたい）
2. **論点 D**（配色モデル）を並行して決める（A と独立）
3. **論点 B**（動線）と **論点 E**（レイアウト）を対にして議論
4. ここまで決まれば **F / G / H / I** は派生で決まる
5. **J / K / L / M** は実装前に確認

---

## 話し合いで詰めたい具体的な問い

以下は議論のきっかけになる具体的な質問です。いずれも即答不要、観点を出す用:

1. 「このアプリを 1 人にだけ薦めるとしたら誰？」 → 論点 A
2. 「使う人は最初の 30 秒で何を見たいか？」 → 論点 B, E
3. 「色を 1 つ変えたときに、次に目が行くべき情報は？」 → 論点 G, H
4. 「現行の 9 カードのうち、明日消えても困らないものは？」 → 論点 I
5. 「スマホで使う場面は想像できるか？」 → 論点 J
6. 「Tailwind を使わないユーザーも想定するか？」 → 論点 D
7. 「既存ユーザーに『学び直し』を強いていいか？」 → 論点 L
8. 「新 UI が動くまで、どれくらい時間をかけていいか？」 → 論点 L, M

## 本書の使い方

- 話し合いで決まった事項は、本書に追記するのではなく **新規文書**（例: `ui-design.md`）に切り出してください
- 本書は「何を決めなきゃいけないか」を忘れないための参照先として残します
- 決まった論点には ✅、保留は 🤔、却下された選択肢は ❌ のマーカーを付けると履歴が残ります
