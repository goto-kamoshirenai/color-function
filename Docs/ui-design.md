# UI 設計（確定事項と派生論点）

[ui-redesign-discussion.md](./ui-redesign-discussion.md) の議論で確定した方針と、そこから派生した新たな検討事項を記録します。

本書は **生きた文書** です。決定が変わった場合は本書を更新し、議論中の項目は 🤔 マーカーで残してください。

---

## 1. 確定した基本方針

### 1.1 ターゲット戦略 ✅

**ペルソナを絞らず、メイン動線だけを特定分野に最適化する** ハイブリッド戦略を採用。

| 位置づけ | 対象 | UI の扱い |
|---|---|---|
| **メイン動線（デフォルト）** | Web 開発関連（P1 Web デザイナー / P2 フロントエンド / P3 a11y） | デフォルトレイアウトで最適化 |
| **サブ動線（カスタム）** | P4 学生 / P5 ブランド / P6 DataViz / その他想定外のユーザー | パネル組み替えで取りこぼしを防ぐ |

### 1.2 モード分岐は設けない ✅

- ロール別モード / 習熟度モード / タスクモードは **採用しない**
- 代わりに **ワークスペースのカスタマイズ性** でユーザーの多様性を吸収
- UI 上は単一、ただし「どのパネルをどこに置くか」はユーザーが決められる

### 1.3 メイン動線 🤔

Web 開発の典型フローを **5 段階** に定義（詳細は [2 節](#2-メイン動線-web-開発フロー)）:

```
[1] 生成・取込  →  [2] 検証  →  [3] 調整  →  [4] 展開  →  [5] 出力
```

**動線の性質**:
- ウィザードで強制せず、任意ステップから出入り可
- どのステップにいても **検証結果が常時可視** であること（調整と検証は切り離さない）
- 「今どのステップか」はユーザーに示す（パンくず or タブ）

## 2. メイン動線（Web 開発フロー）

各ステップの役割と必要機能。現行機能の再配置先も併記。

### Step 1: 生成・取込

**目的**: 出発点となる配色を用意する。

**手段**:
- **A. シード色から自動生成** — 1 色または 2 色を入れて、配色理論（モノクロ/アナロガス/補色/トライアド等）と a11y 制約（AA 以上）で 8 色を埋める
- **B. テンプレートから選択** — 現行の `CardTemplate` 10 種 + マイテンプレート
- **C. 既存資産から取込** — CSV / Tailwind config / Figma Variables JSON / CSS variables をインポート

**今ないもの**: 自動生成エンジン、Figma インポート、マイテンプレート保存

### Step 2: 検証

**目的**: 現在の配色がガイドラインにどれだけ適合しているかを一望する。

**常時可視の情報**:
- WCAG 適合サマリ（AA: X/Y ペア合格、AAA: Z/Y ペア合格）
- CIEDE2000 最小値/平均値（色の区別しやすさの指標）
- 色覚多様性（CVD）シミュレーション適合度

**ドリルダウン**:
- 個別ペアのコントラスト比
- 個別色の相対輝度
- CIEDE2000 マトリクス
- P/D/T 型シミュレーション並列表示

**今ないもの**: 統合サマリ、大テキスト判定 UI、SC 1.4.11 非テキスト判定、CVD シミュ

### Step 3: 調整

**目的**: 不適合な色を修正する。**本アプリの差別化の中核**。

**手段**:
- **自動補正サジェスト** — 「この色をこの値に変えると AA 通る」候補を提示、ワンクリック適用
- **制約付きスライダ** — 色相を固定して明度/彩度だけ動かす。合格ゾーンがグラフ上に可視化
- **ロック機能** — 特定スロット（例: TextA=#000）を固定したまま他を動かす
- **リアルタイムフィードバック** — 色変更の瞬間に全検証結果が更新される

**今ないもの**: すべて新規実装

### Step 4: 展開

**目的**: コア色から派生色を系統的に生成する。

**展開種別**:
- **シェード展開** — Tailwind `50〜950` 形式、Material トーナル、任意ステップ
- **状態色** — hover / active / disabled / focus
- **セマンティック色** — success / warning / error / info
- **ダークモード派生** — 対になるダークバリアント

**今ないもの**: Tailwind シェード自動生成、状態色、セマンティック色、ダーク派生。現行 `ColorExtend` は手動 HSV 線形のみで不十分。

### Step 5: 出力

**目的**: 開発環境に持ち込める形式で書き出す。

**形式**（既存 + 追加）:
| 形式 | 現行 | 追加優先度 |
|---|---|---|
| CSV | ✅ | - |
| Tailwind (JS/TS) | ✅ | シェード出力を拡張 |
| CSS Variables | ✅ | - |
| SASS | ✅ | - |
| JSON | ✅ | - |
| W3C Design Tokens | ❌ | 高 |
| Figma Variables | ❌ | 中 |
| Style Dictionary | ❌ | 中 |

**共有機能**: URL 共有（状態を URL に埋める）も Step 5 に含める。

## 3. 柔軟性レイヤー（パネル組み替え）

メイン動線から外れた用途を取りこぼさないための仕組み。

### 3.1 基本コンセプト

- **すべての機能は「パネル」として実装** し、メインステージに配置可能
- パネルは **追加・削除・移動・サイズ変更** ができる
- ユーザーは現在のレイアウトを **ワークスペース**として保存・復元できる
- 初期レイアウトは Web 開発向けに事前設定されている（= デフォルトワークスペース）

### 3.2 パネルの種類（例）

既存機能 + 新機能をパネル単位に再編成:

| カテゴリ | パネル |
|---|---|
| **エディタ系** | カラーエディタ / 制約スライダ / ロック設定 |
| **視覚化系** | HueWheel / ValueMeter / 相対輝度ゲージ / パレット一覧 |
| **分析系** | WCAG サマリ / WCAG 詳細マトリクス / CIEDE2000 / 色覚多様性シミュ / Entropy |
| **プレビュー系** | UI プレビュー（7 レイアウト）/ 実要素プレビュー |
| **展開系** | Tailwind シェード / 状態色 / ダーク派生 |
| **出力系** | エクスポートパネル / URL 共有 / レポート生成 |
| **補助系** | 生成エンジン / テンプレート / インポート |

### 3.3 ワークスペースプリセット

デフォルト + 用途別プリセット（ユーザーが選べる出発点）:

- **Web 開発（デフォルト）** — 生成 + 検証 + 調整 + Tailwind 出力
- **a11y 監査** — WCAG 詳細 + CVD シミュ + レポート出力
- **学習** — HSV + HueWheel + Entropy + ガイド
- **ブランド提案** — プレビュー + Template + ColorExtend
- **DataViz** — CIEDE2000 + 多色モード（将来）

プリセットはユーザーが増減できる（マイワークスペース保存）。

### 3.4 実装上の要件

- **永続化** — localStorage にレイアウト情報を保存
- **リセット** — 「デフォルトに戻す」ボタン
- **共有** — レイアウト自体を URL や JSON でシェア可能に（副次的）

## 4. 派生する新たな検討事項

基本方針の決定を受けて、詰めるべき論点が変化しました。

### 4.1 🤔 新 論点 N: パネルシステムの技術実装

**問い**: パネルの追加/削除/移動をどう実現するか。

**選択肢**:
| 案 | 方式 | 特徴 |
|---|---|---|
| N1 | react-grid-layout 等の OSS | 実装早い、柔軟、保守コスト中 |
| N2 | CSS Grid + 自前実装 | 軽量、独自仕様で自由だが工数大 |
| N3 | ドッキングパネル系 OSS（react-mosaic 等） | IDE ライクな UX、学習コスト高 |
| N4 | 最小限（並び替えだけ） | ドラッグは後回し、まずは表示順の切替のみ |

**議論のたたき台**: MVP では N4 → v2.x で N1 が現実的。

### 4.2 🤔 論点 D: 配色モデルの再検討

Web 開発メイン動線を前提にすると、配色モデルの選択肢が絞られる:

| 案 | 内容 | Web 開発との親和性 |
|---|---|---|
| D1 現行（4 役 × A/B） | Main/Base/Accent/Text × A/B | ◯ 提案には使えるが Tailwind 流儀と乖離 |
| D2 + 状態色 | D1 + hover/active/disabled/focus | ◎ 実装で直接使える |
| D3 Primary/Secondary + シェード | Material 風 50〜950 | ◎ Tailwind と完全一致 |
| D4 混合 | コア役割（D1 or D3）+ 状態色 + セマンティック色 | ◎◎ 実装網羅性最高、UI 複雑化が課題 |

**方向性**: 現行の 4 役モデルは「提案の語彙」として残しつつ、**展開ステップ（Step 4）で Tailwind シェード形式にも変換できる** 仕組みにするのが穏当。つまり D1 を基盤、D3 を派生として扱う。

議論継続。

### 4.3 🤔 論点 E: レイアウトの具体形

パネル組み替えを前提とすると、論点 E の 5 案は以下のように再解釈される:

- ✅ **E5（Figma 的レイヤー型）** がもっとも方針に合致
- E1/E2 は「デフォルトワークスペース」の一種として実現可能
- E3（ウィザード）は強制しない方針と矛盾
- E4（現行のフリーフォーム）は柔軟性の素地として近い

**決定**: E5 相当のドッキング可能パネル型を採用。初期配置で E1（エディタ型）もしくは E2（ダッシュボード型）風に見せる。

### 4.4 🤔 論点 O（新規）: ステップインジケータの有無

メイン動線を示す視覚要素が必要か。

**選択肢**:
- O1. 上部にタブ（`[1] 生成  [2] 検証  [3] 調整  [4] 展開  [5] 出力`）
- O2. サイドバーに縦型ステップ一覧
- O3. パンくず（現在ステップのみ）
- O4. 設けない（パネル配置で暗示）

**トレードオフ**: O1/O2 は動線を明示できるが、柔軟性レイヤーと干渉する（「ステップ 3 の中にパネルが並ぶ」のか「ステップはただのフィルタ」なのか）。O4 は自由度最大、迷子リスク。

### 4.5 🤔 論点 P（新規）: デフォルトワークスペースの内訳

メイン動線を視覚化するデフォルト画面に、どのパネルをどう配置するか。

ワイヤーフレーム案は次節で仮案を提示するが、議論で詰める必要がある。

### 4.6 据え置きの既存論点

以下は [ui-redesign-discussion.md](./ui-redesign-discussion.md) で未決定のまま継続:
- **論点 F**（色編集 UI の位置） — パネル化に合わせて左ドック常駐が有力
- **論点 G**（検証結果の露出） — 統合サマリパネル + 個別パネルの二層
- **論点 H**（プレビュー） — メインステージの候補、ただし差し替え可
- **論点 J**（モバイル） — パネル組み替えはモバイルと相性悪い、別途判断
- **論点 K**（ダーク/i18n）
- **論点 L**（移行）
- **論点 M**（技術スタック）

## 5. デフォルトワークスペース仮案（叩き台）

あくまで議論のためのスケッチ。**確定案ではありません**。

### 5.1 初期レイアウト（デスクトップ）

```
┌─────────────────────────────────────────────────────────────┐
│ [Header: ロゴ / ステップ: 生成 検証 調整 展開 出力 / 設定] │
├─────────┬──────────────────────────────┬────────────────────┤
│         │                              │                    │
│ カラー  │      プレビュー/ステージ     │  WCAG 合格サマリ   │
│ エディタ│   (現在ステップに応じ変化)   │                    │
│         │                              │  CIEDE2000 サマリ  │
│ [Main]  │                              │                    │
│ [Base]  │                              │  CVD サマリ        │
│ [Accent]│                              │                    │
│ [Text]  │                              │  [詳細パネルへ]    │
│         │                              │                    │
│ 制約    │                              │                    │
│ スライダ│                              │                    │
│         │                              │                    │
├─────────┴──────────────────────────────┴────────────────────┤
│ [補助パネル: Template / 展開 / 出力 / 追加パネル]           │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 ステップ切替時の挙動

- **ステップバー**をクリックで中央ステージの内容が切替
- サイドパネル（左エディタ、右検証）は常時固定
- ユーザーが「パネルの固定を解除」すれば中央に配置可能

### 5.3 機能組み替えの UI

- 右上「レイアウト編集」ボタンで編集モードに入る
- パネルのヘッダをつかんで移動、×ボタンで削除
- 「＋パネルを追加」メニューから一覧選択
- 編集完了で現在レイアウトを保存、または名前付きワークスペースとして別名保存
