# ユーザーペルソナ

本アプリ「Color Follows Function」を **使いたいと思う可能性が高い** 利用者像を 6 パターンに分けて整理します。現行機能（[features.md](./features.md)）とのフィット/ズレを可視化し、抜本的な見直しの判断材料にすることが目的です。

最後に [横断分析](#横断分析--見直しのヒント) で、ペルソナ間で対立する要求や現行アプリの死角をまとめています。

---

## P1. 受託 Web デザイナー（フリーランス）

| 属性 | 値 |
|---|---|
| 名前（仮） | 佐藤 優希 |
| 年齢 | 28 |
| 職能 | Web デザイナー（Figma メイン） |
| 勤務形態 | フリーランス 3 年目 |
| 技術知識 | CSS はある程度書ける。Tailwind は触ったことあり。色理論は独学 |

### 目的・ゴール
- クライアントに提出する配色案を **提案の場で即座に検証** したい（「AA 通りますよ」と言い切るため）
- 自分が感覚で作った配色を **裏付ける数値** が欲しい

### 現状の課題
- Figma のプラグインで都度コントラストを調べているが、4 役 × 2 段階を横断した評価ツールがない
- クライアントに渡すのは Figma のファイルだけで、開発側が実装した結果と色がズレることが多い

### このアプリに期待する価値
- 4 系統 ×2 の配色モデル（Main/Base/Accent/Text × A/B）が **そもそも提案書の構造と一致** している
- PreviewSVG で複数のレイアウトに一気に当てて見せられる

### 優先する機能
1. **Template / ColorExtend** — 提案のたたき台を素早く作る
2. **PreviewSVG** — 営業/会議で画面見せ
3. **Contrast** — 提案書に添付できる数値

### 使わない/不要な機能
- CIEDE2000、相対輝度、Entropy はほぼ見ない（意味が理解できない）

### 抜本見直しの視点
- **「数値の意味がわからないパネルは邪魔」** になり得る。上級者向けパネルは初期非表示 or 別モードに分けた方がこの層には刺さる。

## P2. フロントエンドエンジニア（スタートアップ）

| 属性 | 値 |
|---|---|
| 名前（仮） | 田中 健一 |
| 年齢 | 32 |
| 職能 | フロントエンド / デザインシステム担当 |
| 勤務形態 | スタートアップ（10〜30 名） |
| 技術知識 | React / Next.js / Tailwind を日常的に使用。色はデザイナーと協働 |

### 目的・ゴール
- デザイナーが決めた配色を **Tailwind config にそのまま取り込む**
- PR レビューで「この色は AA 通ってないよね？」を機械的に言えるようにしたい

### 現状の課題
- 色のエクスポート先がバラバラ（Figma variables → JSON → Tailwind 手書き）で食い違いが起きる
- 手元の CLI / スクリプトで都度 WCAG チェックを書くのは面倒

### このアプリに期待する価値
- **CSV / Tailwind JS・TS / CSS variables / SASS / JSON の直接出力**（現行で満たせている）
- 設定を CSV で持ち回して CI に組み込めそう

### 優先する機能
1. **CSV エクスポート（Tailwind config 出力）** — 最大価値
2. **Contrast** — 自動検証のためのロジック/閾値の確認
3. **Template** — 新規プロジェクトの初期配色

### 使わない/不要な機能
- SplashScreen、AboutPanel、ColorExtend（明度バリエーションは Tailwind shade で自動生成する派）

### 抜本見直しの視点
- **CLI 化 / API 化の需要** がある。「設定を URL 共有 → CI から fetch → 違反検出」の経路が無く、毎回画面を開く必要がある。
- Tailwind 出力の構造 `main.a / main.b` が Tailwind 慣習（`primary.50 〜 900` の数値シェード）と異なり、実運用で使いにくい可能性。

## P3. アクセシビリティ専門家（インハウス）

| 属性 | 値 |
|---|---|
| 名前（仮） | 鈴木 美咲 |
| 年齢 | 35 |
| 職能 | アクセシビリティコンサルタント |
| 勤務形態 | 大企業の社内 a11y チーム |
| 技術知識 | WCAG 2.1/2.2 を一字一句覚えている。APCA も追っている |

### 目的・ゴール
- 既存プロダクトの配色監査レポートを作る
- WCAG AA/AAA 判定の **根拠となる数値** と、大きなテキスト対応の内訳を記録する

### 現状の課題
- 監査対象の配色を一括で投入して、組合せ全数を一覧する手段が弱い
- 「大きなテキストだと AA 通るが通常は NG」といった条件分岐を人力で考えたくない

### このアプリに期待する価値
- コントラスト比 + AA/AAA 判定が 1 画面で一覧できる
- 相対輝度（raw 値）が出るので根拠提示できる

### 優先する機能
1. **Contrast** — 命綱
2. **RelativeLuminance** — 根拠資料
3. **CSV エクスポート** — レポート添付

### 使わない/不要な機能
- Template、ColorExtend、PreviewSVG、Entropy、HSV はほぼ不要

### 抜本見直しの視点
- **大きなテキスト判定が UI に露出していない**（`getWCAGLevel` は `isLargeText` を受けるが、`CardContrast` は常に通常テキスト扱い）。この層には致命的な不足。
- **APCA（WCAG 3.0 ドラフト）対応**への関心が高い。今後の主戦場になり得る。
- レポート出力（PDF / Markdown）の要求が強い。

## P4. デザイン系学生（色彩理論の学び）

| 属性 | 値 |
|---|---|
| 名前（仮） | 山田 翔太 |
| 年齢 | 20 |
| 職能 | 美大 / 情報デザイン学科 2 年 |
| 技術知識 | HTML/CSS を授業で触り始めた段階。色彩検定を勉強中 |

### 目的・ゴール
- 授業課題で「なぜこの配色が落ち着いて見えるのか」を **数値で説明** したい
- 色相環・HSV の感覚を身につけたい

### 現状の課題
- 教科書の色相環と実際のピクセル色が結び付かない
- Adobe Color など既存ツールは「きれいなパレット提案」はあるが「なぜ？」を説明してくれない

### このアプリに期待する価値
- HueWheel + ValueMeter で色相・彩度・明度を視覚化してくれる
- Entropy カードが「統一感/バランス/カラフル」と日本語で評価してくれる（教育的）

### 優先する機能
1. **HSV** — 学習の入り口
2. **Entropy** — 評価の言語化
3. **CIEDE2000** — 「2 色が違って見えるか」の数値化

### 使わない/不要な機能
- 現時点では CSV エクスポートは使わない（課題提出は画像で十分）
- Tailwind 出力は知らない

### 抜本見直しの視点
- **学習者に最適化されたモード**（各指標の意味を初めに解説する、クイズ形式など）があれば強力な教育ツールになる。現行はいきなり数値が並んでいて学習導線が弱い。
- 色彩検定の用語（PCCS トーン、補色、類似色）との紐付けがあると学習価値が跳ね上がる。

## P5. ブランド/グラフィックデザイナー

| 属性 | 値 |
|---|---|
| 名前（仮） | 中村 由美 |
| 年齢 | 40 |
| 職能 | ブランドデザイナー（ロゴ・VI 中心） |
| 勤務形態 | デザイン事務所 |
| 技術知識 | Illustrator / Photoshop が主戦場。Web の事情には明るくない |

### 目的・ゴール
- 企業 VI のカラーシステムを **意味付けて提案** する（「主役・背景・強調・文字」）
- クライアント前でその場で微調整して見せたい

### 現状の課題
- Web と違って印刷もあるため CMYK との整合性も気になる
- 配色の **哲学** を言語化する手段が少ない（「洗練されていてアクセントが効いている」を数値で）

### このアプリに期待する価値
- 4 役 × 2 段階のモデルが **ブランドガイドラインの語彙** と近い
- Entropy の「統一感/バランス/カラフル」がクライアント説明に使える

### 優先する機能
1. **Template** — 参照バリエーション
2. **PreviewSVG** — 実装後のイメージ共有
3. **ColorExtend** — 微調整の幅探し

### 使わない/不要な機能
- WCAG は意識するが副次的。CSV エクスポートは不要
- CMYK 未対応なのでそこは外部ツール（Adobe）と往復

### 抜本見直しの視点
- **印刷/CMYK ワークフローへの橋渡し** が欠けている。RGB 専用ツールと割り切るのか、ブランド系まで取るのかは戦略判断。
- **配色の意味付けラベル**（「プライマリ/セカンダリ/…」の再定義、タグ、メモ欄）があれば提案書に直結する。

## P6. データビジュアライゼーション設計者

| 属性 | 値 |
|---|---|
| 名前（仮） | 高橋 啓介 |
| 年齢 | 36 |
| 職能 | BI ダッシュボード設計 / データサイエンティスト兼任 |
| 技術知識 | D3.js / Plotly / Observable。色覚多様性（CVD）にも配慮 |

### 目的・ゴール
- ダッシュボードで **8〜12 色を同時に使う** ときに、どの色も区別可能にしたい
- Protanopia / Deuteranopia でも破綻しない配色を検証

### 現状の課題
- カテゴリカルカラーパレットの検証ツールはあるが、`ΔE`（知覚距離）を横断で見られるものが少ない
- 「4 役 × 2 段階」モデルだと数が足りない

### このアプリに期待する価値
- **CIEDE2000 マトリクス** が最大の魅力。8 色総当たりで知覚差を視覚化できる
- chroma-js を使っているので拡張余地がありそう

### 優先する機能
1. **CIEDE2000** — 唯一無二
2. **RelativeLuminance** — 白背景/黒背景での視認性検証
3. **HSV** — カテゴリ間の色相等間隔性の確認

### 使わない/不要な機能
- Template は用途不一致（ブランド配色は不要、12 色のシリーズ型が欲しい）
- PreviewSVG は UI レイアウト想定なので DataViz には合わない

### 抜本見直しの視点
- **8 色固定の配色モデル**が足かせ。**任意数のパレット**（3色〜20色）を扱えるモードがあれば DataViz/科学ビジュアライゼーション市場を取れる。
- **色覚多様性シミュレーション**（シミュレート版を隣に並べる）は技術的に chroma-js で可能。現行は未実装。

## 横断分析 / 見直しのヒント

### 機能 × ペルソナ マトリクス

| 機能 | P1 デザイナー | P2 エンジニア | P3 a11y | P4 学生 | P5 ブランド | P6 DataViz |
|---|---|---|---|---|---|---|
| Template | ◎ | ○ | ✕ | ○ | ◎ | ✕ |
| ColorExtend | ◎ | ✕ | ✕ | ○ | ◎ | ○ |
| PreviewSVG | ◎ | ○ | ✕ | ○ | ◎ | ✕ |
| Contrast | ○ | ◎ | ◎ | ○ | ○ | ○ |
| RelativeLuminance | △ | ○ | ◎ | △ | △ | ○ |
| CIEDE2000 | ✕ | △ | △ | ○ | △ | ◎ |
| Entropy | △ | ✕ | ✕ | ◎ | ○ | ○ |
| HSV | △ | △ | ✕ | ◎ | ○ | ○ |
| CSV 出力 | ○ | ◎ | ◎ | ✕ | △ | ○ |

凡例: ◎ 主戦力 / ○ 使う / △ たまに / ✕ ほぼ使わない

### 観察できる対立軸

1. **「きれいな配色を提案したい」 vs 「数値で厳密に検証したい」**
   P1/P5 は視覚的提案、P2/P3 は数値検証。現行は両者を同じ 1 画面に並べており、どちらの層にも中途半端感が残る。
   → **モード分離**（Design / Engineering / a11y の 3 プリセット）が効きそう。

2. **「8 色固定モデル」 vs 「任意数パレット」**
   P1–P5 は 4 役 × 2 段階で十分。P6 の DataViz は完全に不一致。
   → 別アプリに切り出すか、**パレットサイズ可変モード**を追加するか。

3. **「Web 専用」 vs 「印刷/CMYK」**
   P5 のみが CMYK を求めるが、対応コストは高い。スコープ外と明示する方が潔い。

4. **「画面で使う」 vs 「CI/CD に組み込む」**
   P2 は CLI / API を望む。ブラウザアプリの枠を超える判断が必要。

### 現行アプリの死角

- **大きなテキスト判定のトグルが UI に無い** — P3 にとって必須機能が隠れている（[wcag.md](./wcag.md) 参照）
- **APCA / WCAG 3.0 ドラフトへの対応なし** — 今後 1〜2 年で業界標準が動く可能性
- **色覚多様性シミュレーション未実装** — P3/P6 が強く望む
- **配色の意味・メモを保存する機能なし** — P5 が言語化できない
- **URL 共有 / 永続化なし** — 全ペルソナに共通する軽い痛み（作業を続行できない）
- **学習導線なし** — P4 が迷子になる
- **ダークモード対応の UI プレビュー** — PreviewSVG が現状はライト/ダーク両方を想定していない

### 戦略的な問い（Product 判断用）

1. **誰を第一の顧客にするか？** 現状、機能は全方位だが訴求が全方位になっている
2. **単独ツールか、ワークフローの一部か？** ブラウザで完結/CLI 化/プラグイン化
3. **無料公開か、サブスク/フリーミアムか？** 教育層（P4）は無料、企業 a11y（P3）は有償レポートが成立
4. **どの規格をサポートするか？** WCAG 2.x 固定 or APCA 追従
5. **オフライン/プライバシー要件は？** 色データが外部送信されないことは現状のアピールポイントになる
