# 必要機能の洗い出し

## 本アプリのコア価値（前提）

ユーザーが定義した 4 本柱:

1. **ビジュアライズ** — 色における数値を可視化する
2. **ガイドライン適合** — 各種ガイドラインへの適合判定と、適合するための調整
3. **配色の自動生成** — 最低限の技術ラインを満たした配色を自動で作る
4. **テーマカラーからの展開** — 主要色から派生色を展開する

本ドキュメントは、この 4 本柱を満たすために **どんな機能が必要か** を網羅的に洗い出し、現行機能とのギャップを示します。優先度付けは末尾の [ロードマップ](#ロードマップ優先度) にまとめます。

関連: [features.md](./features.md)（現行機能）・[personas.md](./personas.md)（想定ユーザー）・[wcag.md](./wcag.md) / [calculations.md](./calculations.md)（計算仕様）

## 1. ビジュアライズ（数値の可視化）

**目的**: 色が持つ数値属性を、直感と分析の両方で理解できる形で見せる。

### 1.1 色空間別の数値表示

| 色空間 | 必要な理由 | 現行 |
|---|---|---|
| HEX / RGB | 最も普遍的な表現 | 部分的（HEX のみ）|
| HSV | 直感的調整（H/S/V） | ✅ HSV カード |
| HSL | CSS 標準・Tailwind 整合 | ❌ |
| LAB | 知覚均等・CIEDE2000 の元 | ❌ |
| LCH / OKLCH | 近年の主流、CSS Color Level 4 | ❌ |
| CMYK | 印刷向け参考値 | ❌ |

**必要機能**: 各色スロットで、複数色空間の値を同時表示（クリックでコピー可能）。

### 1.2 単色のビジュアライズ

- **色相環上の位置** — `HueWheel`（現行◯）
- **彩度 × 明度 グラフ** — `ValueMeter` / `CardEntropy` の 2D プロット（現行◯）
- **相対輝度ゲージ** — 0〜1 のバー表示（現行は数値のみ、ゲージ化が望ましい）
- **ΔE 近傍色ヒートマップ** — その色から ΔE が小さい順に候補提示
- **色温度** — ケルビン近似値
- **CSS で使える名前候補** — 近傍の CSS Named Color / Tailwind 既定色（「Tailwind で言う `sky-500` に近い」等）

### 1.3 複数色のビジュアライズ（パレット分析）

- **ΔE00 マトリクス** — 全色総当たり（現行◯）
- **色相分布円** — 色相環上に配色点を配置（現行◯）
- **明度ヒストグラム** — パレットの明度偏りを検知
- **色相ホイール軌跡** — アナロガス/補色/トライアド等の配色理論図形との重ね合わせ
- **エントロピー指標** — hueVariety / saturationUnity / lightnessBalance（現行◯、ただし[指標の設計見直し](./calculations.md#6-実装上の既知の特性要注意点)が必要）

### 1.4 色覚多様性シミュレーション（新規）

- **P型（Protanopia）/ D型（Deuteranopia）/ T型（Tritanopia）** の見え方プレビュー
- **全体モノクロ** シミュレーション
- **サイドバイサイド表示**: 通常 ↔ シミュレート版を並べて比較
- chroma-js の色変換だけでは実現できないため、シミュレーション行列（Brettel/Viénot 等）を自前実装 or 外部ライブラリ導入

### 1.5 UI プレビュー（使用感のビジュアライズ）

現行 `CardPreviewSVG` の 7 レイアウトは妥当。拡張候補:

- **ライト/ダークモード切替**
- **実要素プレビュー**: ボタン（enabled/hover/active/disabled/focus）、リンク（visited 含む）、フォーム、アラート
- **実サイト貼付**: URL を入れると iframe 上に配色オーバーレイ（技術的には難しい。スクリーンショット読込 + 色置換でも可）
- **型崩れ検出**: テキストサイズが AA 未達の要素を警告

## 2. ガイドライン適合判定と調整

**目的**: 「どの規格で、どこが通るか/通らないか」を一目で把握し、通すための具体的アクションを提示する。

### 2.1 対応すべき規格

| 規格 | 優先度 | 内容 | 現行 |
|---|---|---|---|
| WCAG 2.1/2.2 SC 1.4.3 (AA) | 必須 | 通常テキスト ≥4.5:1 / 大≥3.0:1 | ✅（ただし大テキスト UI なし） |
| WCAG 2.1/2.2 SC 1.4.6 (AAA) | 必須 | 通常 ≥7.0:1 / 大 ≥4.5:1 | ✅ |
| WCAG 2.1/2.2 SC 1.4.11 (非テキスト) | **不足** | UI 部品・グラフィック ≥3:1 | ❌ |
| WCAG 2.2 SC 1.4.12 (Text Spacing) | 参考 | 文字間・行間（配色の範疇外） | - |
| WCAG 3.0 / APCA（ドラフト） | 強く推奨 | 人間の知覚に基づく新基準（Lc 値） | ❌ |
| 色覚多様性対応（JIS Z 8210 他） | 推奨 | P/D/T 型でも区別可能な配色 | ❌ |
| Material Design (Tonal) | 任意 | トーナルパレット整合 | ❌ |
| ブランドガイドライン | 任意 | カスタム閾値（企業独自 AA 相当） | ❌ |

### 2.2 判定機能

- **大テキスト判定トグル** — カードレベルの切替（必須）
- **UI 部品コントラスト判定** — 非テキスト要素の 3:1 判定
- **パス/不達サマリ** — 「AA: 合格 6/8・不合格 2/8」をヘッダに集約
- **組合せフィルタ** — 「テキスト×背景」「ボタン色×背景」「アクセント×背景」など用途別ビュー
- **規格の併記** — 1 ペアに対し WCAG AA / AAA / APCA Lc 値を同時表示

### 2.3 適合のための調整支援（差別化ポイント）

単なる判定ツールではなく「どう調整すれば通るか」を示す。

- **最小補正サジェスト** — 「AA に達するには明度を ΔL ≥ 0.12 だけ暗くしてください」等、元色から最も近い合格色を提案
- **ワンクリック修正** — サジェストに従って色を自動置換
- **制約付きスライダ** — 色相を固定、明度・彩度だけ動かすスライダで合格ゾーンを可視化（グリーンバンド表示）
- **固定ロック** — 「Base は動かさず Text 側を調整」モード
- **リアルタイムガイドレール** — 色変更時にその場で「AA 境界」を可視化
- **一括自動調整** — パレット全体を、規定の対比関係を保ったままアクセシブルに寄せる

### 2.4 レポート出力

- **Markdown / PDF レポート** — 配色一覧 + 合否 + 根拠数値
- **SC（Success Criteria）別の結果** — 監査用
- **a11y 検証ツール（axe/Lighthouse）向け JSON**

## 3. 配色の自動生成

**目的**: 「最低限の技術ライン（= ガイドライン適合）を満たす」配色を、キーワードや制約から自動生成する。

### 3.1 生成のインプット

ユーザーが指定できる条件:

- **シードカラー**（1 色〜複数色）
- **配色理論**: モノクロマティック / アナロガス / 補色 / スプリットコンプリメンタリ / トライアド / テトラード
- **ムード/キーワード**: warm / cool / earthy / playful / corporate / premium 等
- **用途**: ダッシュボード / ブログ / EC / ランディング / ブランド（Preview カードと連動）
- **アクセシビリティ制約**: AA / AAA / APCA Lc ≥ X
- **色覚多様性**: P/D/T 型でも区別可能
- **ロック**: 特定スロット（例: Text=#000 固定）は動かさない

### 3.2 生成エンジン（制約ソルバ）

- **制約充足型**: 「MainA = シード、Base は白、Text は Main に AA 以上、Accent は Main と色相差 120°」のような多制約を満たす解を探索
- **ランダマイズ** + シード再現性（URL に seed 値を含める）
- **複数候補提示**: 1 ボタンで 5〜10 候補を並列表示
- **評価スコア**: 生成候補ごとに WCAG 合格数 / ΔE 広さ / CVD 対応度をスコアリング

### 3.3 既存テンプレートとの融合

- 現行の 10 種テンプレート（`colorConst.ts`）を「スターターセット」として保持
- ユーザーが生成した配色を **マイテンプレート** として保存
- コミュニティテンプレート（将来）

### 3.4 インポートからの補完

- 既存のブランド 1〜2 色から 8 スロット全てを埋める
- Adobe Color / Coolors / Figma variables の JSON をインポート → 不足スロットを自動補完

## 4. テーマカラーからの展開

**目的**: 主要色（テーマカラー）から実装で必要な派生色を系統的に展開する。

### 4.1 シェード展開（濃淡）

- **Tailwind シェード**（50, 100, 200, …, 900, 950）自動生成
- **Material Design トーナルパレット**（tone 0〜100）
- **任意ステップ**（例: 5%, 10%, 15% 刻み）
- **視知覚的に等間隔**（LCH/OKLCH 基準）— 単純な HSL 線形補間よりも自然

### 4.2 状態色（State Colors）展開

UI 実装で必要:

- **hover** — 元色から明度を 1 段階シフト
- **active** — もう 1 段階シフト
- **disabled** — 彩度を下げ、輝度を中間寄りに
- **focus ring** — 元色の補色系 or 高コントラストバリアント

### 4.3 セマンティックカラー展開

- **success / warning / error / info** — テーマカラーと整合する色相を自動選定
- ブランド色相から離しすぎず、信号色として認識できるギリギリを狙う

### 4.4 ダークモード派生

- **自動反転** — 背景/前景を入れ替えたバリアント
- **トーン調整** — 単純反転ではなく、ダーク UI で推奨される「彩度控えめ・明度中間」にシフト
- **コントラスト保存** — ライト版と同じ比率のコントラストをダーク版で再現

### 4.5 展開の制約

- どの派生色も **元のコア規格（AA 等）に準拠** する
- 派生色間でも **ΔE が一定以上** 確保される（Tailwind シェードが隣同士で区別できる）
- 現行の `ColorExtend` は手動・HSV 線形のため、この目的には不足

## 5. 横断機能（Cross-cutting）

### 5.1 永続化・共有

- **URL 共有** — 配色状態を URL のクエリ/ハッシュに埋め込む（ログイン不要）
- **localStorage 永続化** — 再訪時に前回の配色を復元
- **アカウント/クラウド保存** — チーム共有想定
- **バージョン履歴** — Undo/Redo、名前付きスナップショット

### 5.2 エクスポート拡張

現行の CSV / Tailwind (JS/TS) / CSS / SASS / JSON に加え:

- **W3C Design Tokens**（`.tokens.json`）
- **Style Dictionary** 入力形式
- **Figma Variables JSON**
- **Adobe Color .ase / .aco**
- **iOS Asset Catalog** (Contents.json)
- **Android colors.xml**
- **SVG スウォッチパレット画像**
- **PDF レポート**（プレビュー + 数値 + 合否）
- **Tailwind シェードフル出力**（`primary.50 〜 primary.950`）

### 5.3 インポート拡張

- Figma Variables JSON
- Adobe Color .ase
- CSS/SASS ファイル読込
- Tailwind config から既存配色を抽出

### 5.4 学習・ヘルプ

- **ガイドツアー** — 初回アクセス時に各カードの意味を解説
- **数値の意味説明** — ΔE・エントロピー等を図解
- **規格解説** — WCAG SC 1.4.3 とは何か、AA と AAA の違い
- **配色理論ミニ講座** — 補色/類似色/トライアド等

### 5.5 UI/UX

- **ダークモード対応**（アプリ自身の UI）
- **キーボードショートカット**
- **レスポンシブ対応** — モバイル閲覧
- **アクセシブルな UI** — 自身が WCAG AA 準拠

### 5.6 多言語

現行: 日本語・英語。追加候補:
- 中国語（簡体/繁体）
- 韓国語
- 英語圏以外の市場視野

### 5.7 統合・外部連携

- **Figma プラグイン** — Figma 内で配色生成・検証
- **VS Code 拡張** — CSS/Tailwind 内の色に対するリアルタイム a11y 警告
- **CLI** — CI に組込める `npx color-function check ./tailwind.config.ts`
- **REST / JSON API** — 外部ツールから配色検証

## 6. 現行機能とのギャップサマリ

| 現行機能 | 対応する柱 | 状態 |
|---|---|---|
| `MyColorPanel`（4 役×2 段階） | 全体 | ✅ 基盤として機能 |
| `CardHSV` / `HueWheel` / `ValueMeter` | 1 ビジュアライズ | ✅ 単色視覚化は OK / HSL・LAB 等は未対応 |
| `CardRelativeLuminance` | 1, 2 | ✅ 数値提示 / ゲージ視覚化はなし |
| `CardCIEDE2000` | 1 | ✅ 8 色マトリクスは特徴 |
| `CardEntropy` | 1 | △ 指標の設計に改善余地 |
| `CardContrast` | 2 | ⚠️ 大テキスト UI 欠落・SC 1.4.11 未対応・APCA 未対応 |
| `CardTemplate`（10 種） | 3 | △ 静的、生成エンジンなし |
| `ColorExtend` | 4 | △ 手動 HSV 線形のみ、Tailwind シェード/状態色/ダーク派生なし |
| `CardCSV`（6 形式） | 横断 | ✅ エクスポート基盤は十分 / Design Tokens 等は未対応 |
| `CardPreviewSVG`（7 レイアウト） | 1 | ✅ / ダークモード・実要素プレビューは未対応 |
| `AboutPanel` / `HelpXxx` | 横断 | △ 情報提示のみ、学習導線は弱い |

**大きな穴**:
1. **色覚多様性対応が皆無**（1, 2 両方の柱にまたがる欠落）
2. **適合のための調整支援が皆無**（2 の柱の半分が欠けている）
3. **配色の自動生成エンジンが存在しない**（3 の柱そのものが未実装、テンプレートは静的）
4. **テーマ展開が単色のバリエーション表示で止まっている**（4 の柱が弱い）
5. **URL 共有/永続化が無く作業の継続性が無い**（横断）

## 8. 設計上の判断が必要な論点

実装に入る前にユーザー判断が必要な項目:

1. **4 役×2 段階モデルを維持するか** — 第一顧客が DataViz 設計者なら拡張が必要
2. **APCA をいつ取り込むか** — WCAG 2.2 との併存 or 併記
3. **色覚多様性シミュレーションの実装方針** — 自前で行列実装 / 外部ライブラリ（colorblind, color-blind）
4. **配色生成エンジンの実装方針** — 制約ソルバを自前で書く / 既存（colorjs.io, culori 等）に寄せる
5. **URL 共有形式** — クエリ? ハッシュ? 圧縮? どこまで複雑な状態を埋めるか
6. **レポート出力の生成方法** — サーバレス（PDF 生成は重い）/ クライアント（jspdf）/ Markdown 中心
7. **ダークモード（アプリ UI）対応の優先度** — 色選定ツール自身のコントラスト保全
8. **Tailwind v4（CSS-first）対応** — 出力形式が v3 と変わる可能性

これらを踏まえて、v2 の MVP スコープを確定させてから実装に着手することを推奨します。
