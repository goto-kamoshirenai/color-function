# 0001. ペア検証モードの FG/BG パネルを3段階レスポンシブで常時表示する

- ステータス: 採択
- 日付: 2026-06-13

## 文脈

ペア検証モード（`単位=pair` × `ビュー=verify`）では、文字色(FG)と背景色(BG)の関係を縦に長いカード群（WCAG コントラスト・ΔE・CVD・APCA・内訳）で検証する。FG/BG を選ぶ「文字色と背景色パネル」(`PairRolePicker`) はマストヘッドとカード群の間にあり、**スクロールすると画面外へ消える**。結果カードを読みながら FG/BG を切り替えたい、という検証作業の中心動作と噛み合っていなかった。

このパネルをスクロールに追従させて常時手元に置きたい。ただしレイアウト制約は画面幅で大きく変わる:

- メインのコンテンツ列は `max-w-[900px]` の中央寄せ。広い画面では左右にガター（余白）が生まれる。
- スクロールはページ全体ではなく `main` 要素の内側で起きる（ヘッダー56px は固定、フッターのパレットバーも固定）。

## 決定

単一のレスポンシブ戦略ではなく、画面幅で **3つの異なる配置** を使い分ける。

1. **PC (≥1280px)** — 左ガターに `position:fixed`（ヘッダー直下・左上寄せ）で常時固定。コンテンツ列（中央900px）には被らない。
2. **タブレット (640–1279px)** — `main` 最上部に `position:sticky; top:0` で配置し、**通常フローでスペースを占有**する（コンテンツと被らせない）。
3. **スマホ (<640px)** — 常時表示スペースがないため初期は折りたたみ。**両状態で同一の常設ヘッダ**（FG·BG タグ＋現在色チップ＋ラベル＋シェブロン）を出し、行全体をトグルにする。展開するとヘッダの下に細い下罫線を挟んで FG/BG セレクタが現れる（シェブロンの向きだけ反転）。開閉でレイアウトが大きく変わらないよう、情報(?) ボタンはスマホでは出さない（PC/タブレットのみ）。折りたたみ状態のみ `localStorage` に永続化する。

パネルの中身は FG/BG セレクタ2つのみに絞る（従来の見出し・説明文は持たせず、説明は「?」ツールチップへ退避）。

### 1280px という境界の根拠

PC モードの「ガター固定」は、パネルがコンテンツ列に被らないだけのガター幅があって初めて成立する。コンテンツは中央寄せ `max-w-900` なので、ガター片側 = `(viewport − 900) / 2`。縦積みの FG/BG パネルに必要な幅を約180–190px とすると:

```
(viewport − 900) / 2 ≥ ~190  ⟹  viewport ≥ ~1280
```

この算術が Tailwind 標準の `xl`(1280) とほぼ一致したため、独自ブレーク点を増やさず `xl` を採用した。下限の `sm`(640) も Tailwind 標準のスマホ境界に合わせた。

## 代替案

- **全幅で常に sticky 上部バー** — 実装は単純だが、PC でコンテンツ上部を常に専有して縦領域を圧迫し、左右の広い余白を活かせない。
- **全サイズで FAB（フローティングボタン）→ オーバーレイ** — 省スペースだが、現在の FG/BG が一目で見えず、検証中の「今どの2色か」を常時把握したい要求に反する。
- **全サイズで折りたたみトグル** — 一貫はするが、常時表示できる PC/タブレットでもひと手間増え、「余白があるなら常に見せる」要求と相反する。

## 帰結

- ブレーク点（1280 / 640）はガター算術に依存する。コンテンツ列の `max-w` を変えるとガター幅が変わり、PC モードの成立条件が崩れるため、`max-w` 変更時はこのブレーク点も見直すこと。
- CSS 駆動の3段階レイアウトと、JS state（`localStorage`）による折りたたみが共存する。折りたたみ state が効くのは <640 のみで、≥640 では無視して常時展開する必要がある。
