# 09. デザインブリーフ（UI作成ハンドオフ）

> ステータス: ドラフト / 最終更新: 2026-06-10
> 用途: **外部のデザインAI／デザイナーにそのまま渡せる、単体で完結した依頼書**。
> 他ドキュメントを読まなくても理解できるよう自己完結で記述する。

---

## 0. 依頼の要約

**「Color Follows Function」** という Web アプリの UI をデザインしてほしい。
これは配色を「感覚」ではなく「数値」で扱うための、デザイナー／フロントエンド開発者向けの**配色の検証・設計支援ツール**。
本書はそのUI設計に必要な情報を一式まとめたもの。**画面構成・主要部品・インタラクション・トーン・成果物**を定義する。

---

## 1. プロダクトの本質（デザイン判断の拠り所）

- 名前の由来は "Form Follows Function"（形態は機能に従う）のもじり。**「色も機能に従って選ばれるべき」**という思想。
- コアコンセプト: **感覚的に扱われがちな「色」を、数値的・定量的に可視化する**。
- ゆえにこのアプリの主役は**色そのものと、その数値（コントラスト比・色相・輝度・色差など）**。UIの装飾ではない。
- トーン: **精密・信頼できる・データ駆動**。計測器／分析ダッシュボードの落ち着きに近い。可愛らしさや過度な演出は不要。

### ユーザー
- Webデザイナー（配色設計・UIトーン決め）
- フロントエンド開発者（実装時の色の妥当性確認）
- デザインレビュアー（「なぜこの色か」を数値で説明したい）
- 共通項: **色を主観でなく数値で扱いたい人**。専門知識がなくても指標の意味が分かること。

---

## 2. 最重要のデザイン制約（色ツール特有）

1. **UIクロムはモノクロ・中立をベースに、アクセントは1点のみ**
   分析対象の色を正しく知覚させるため、画面の地・枠・文字などUI自体の色は**グレースケールをベース**にする。
   彩度の高い色で画面を染めると、ユーザーが評価中の色の見え方が歪む。**色の主役はユーザーの配色**。
   ただし**唯一のアクセント色**として、ユーザーが配色パレットで指定した1色を `--accent` 変数経由で**アプリ全体に適用**する
   （フォーカスリング・選択・アクティブ強調などの差し色）。コントラスト不足時は明度を自動補正して可読性を保つ（→ [10 §1.1](./10_design_reference.md)）。
2. **アプリ自身がアクセシブルであること**
   アクセシビリティを評価するツールが不適切なコントラストやキーボード非対応では本末転倒。
   十分なコントラスト、フォーカス可視、キーボード操作、適切なARIAを満たす。
3. **ライト／ダーク両対応**
   どちらの地でも色がどう見えるかは重要。両モードを中立なグレースケールで用意する。
4. **リアルタイム性**
   色を編集すると、すべての数値・可視化が**即座に**更新される体験を前提に設計する。

---

## 3. 画面構成（情報アーキテクチャ）

画面は **常設の「配色パレットバー」（下部）** と **「分析・編集領域」（メイン）** の2層が基本。

```
┌───────────────────────────────────────────────┐
│ ヘッダー（薄め・任意：ロゴ／ページナビ／ヘルプ）       │
├───────────────────────────────────────────────┤
│                                               │
│  メイン領域（分析・編集カードを縦スクロール）          │
│  現在のモードに該当するカードだけが並ぶ               │
│                                               │
│    ┌─ コントラスト ─────────┐                    │
│    │  4.53 : 1   AA ✓ AAA ✕ │   ← カード         │
│    └────────────────────────┘                    │
│    ┌─ HSV ──────────────────┐                    │
│    │  H 210°  S 80%  V 60%   │                    │
│    └────────────────────────┘                    │
│                          ↓ スクロール              │
│                                               │
├───────────────────────────────────────────────┤
│ 配色パレットバー（常設・下部・コンパクト）            │
│  [単位 単色|ペア|パレット]  [観点 検証|設計]   [＋]  │
│  ● ● ● ● ●   ← 色スウォッチ（選択／役割を表示）      │
└───────────────────────────────────────────────┘
```

- **パレットバー**はほぼ全ページで常に表示。アプリの「色の保持・操作」の拠点。できるだけコンパクト。
- **メイン領域**は、選択中のモードに対応する分析カードをリアルタイム描画する場。
- パレットの色は**アプリ全体で共有される一つの状態**（どのカードも同じ色を見ている）。

---

## 4. パレットバー（最重要コンポーネント）

下部に常駐するコンパクトなバー。以下を含む。

### 4.1 色スウォッチ列
- パレットの色を横並びのスウォッチで表示（順序つき）。
- スウォッチをクリックで**選択**。選択状態が見て分かること。
- 「ペア」モードでは2つの色に **前景(FG)／背景(BG) の役割バッジ**を表示。役割の入替操作も用意。
- 「＋」で色を追加。各スウォッチから削除・並べ替えができる。
- スウォッチには HEX 値などの簡易ラベルを併記（コンパクトさと両立する範囲で）。

### 4.2 モード選択UI（2軸）
バー内に2つのトグルを置く。**この2軸の組み合わせがメイン領域の表示内容を決める**。

```
単位:  [ 単色 ] [ ペア ] [ パレット ]   ← どの粒度で見るか
観点:  [ 検証 ] [ 設計 ]               ← 何の目的で見るか
```

- **単位**: 単色＝1色の性質、ペア＝2色の関係（主にコントラスト）、パレット＝集合全体。
- **観点**: 検証＝既存配色が妥当かを診断、設計＝良い配色を作る支援。
- 切り替えても**色や選択は失われない**（連続した作業空間）。トグルは現在値が明確に分かる表現に。

### 4.3 レスポンシブ
- 画面が狭いときバーは畳める／スクロールできること。モバイルでも下部常駐の発想は維持。

---

## 5. メイン領域 ＝ 分析カードのスクロール

- メイン領域は**カードの縦スクロール**。現在のモードに該当するカードだけを並べる。
- カードは「1つの指標＝1カード」。今後カードは増えていく前提（拡張可能なレイアウト）。

### カードの構成（共通アナトミー）
各カードに共通して必要な要素:
- **タイトル**（指標名。例: コントラスト比 / HSV / 色差）
- **主たる数値・可視化**（大きく・読みやすく。これがカードの主役）
- **判定・補助情報**（例: AA/AAA バッジ、目安の説明）
- **ヘルプの導線**（指標の意味をその場で学べる。アイコン→ツールチップ／ポップオーバー等）
- **コピー導線**（HEX/HSL 等の値を個別コピー）

### カードの種類（モード別の例。デザインはこれらを横断して成立させる）
- 単色×検証: RGB/HEX、HSL、HSV、相対輝度、対 白/黒 のコントラスト、最寄り色名、色相環
- ペア×検証: WCAGコントラスト比（AA/AAA判定）、色差(ΔE)、テキスト可読性プレビュー、色覚シミュレーション
- パレット×検証: 全色ペアのコントラスト／色差マトリクス、色相分布、SVG・UIプレビュー
- 設計（単位を問わず）: 調和スキーム生成、トーン展開、相手色提案、テンプレート適用

> カードのビジュアルは**指標ごとに最適化**してよい（メーター、色相環、マトリクス表、プレビュー面など）。
> ただし全カードが同じ「タイトル＋主役の数値＋ヘルプ＋コピー」の骨格を共有して並んで見えること。

---

## 6. 主要インタラクション

1. **どこからでも編集**: スウォッチ・カード内ピッカー・数値入力のいずれからでもグローバルな色を編集できる。編集は全カードに即時反映。
2. **カラーピッカー**: 色の追加・編集に使う。HEX 直接入力も可能に。
3. **確認ダイアログ**: 通常の微調整は即時反映だが、**破壊的操作**（全消去、テンプレートでの一括置換など）は反映前に確認を挟む。
4. **共有リンク**: 現在の配色を URL として共有・復元できる（保存先サーバを持たない設計のため、URLが共有の要）。
5. **値コピー**: HEX/HSL などをワンクリックでコピー。

---

## 7. ビジュアル方針（ガードレール内で自由に）

- **UIの地・枠・文字はモノクロ（グレースケール）ベース**。彩度のある色は**ユーザー指定のアクセント1点**に集約し、`--accent` で差し色として使う（§2-1）。
- 数値・ラベルは**等幅 or 可読性の高い数字表現**で、計測器的な信頼感を出す（モック実装は Archivo＋Geist Mono）。
- 余白とグリッドで「整理された分析画面」を作る。情報は多いが、混雑させない。
- スウォッチや色相環など、**色そのものを見せる面は色を歪めない地（中立背景・適切な境界）**に置く。
- アニメーションは状態変化の理解を助ける範囲に留める（リアルタイム更新が主役）。
- ロゴ／ブランド表現は最小限のヘッダーに。アプリ名「Color Follows Function」の思想（精密・機能的）に沿う。

> 美的方向性（タイポ・余白感・装飾の度合い）はあなたの裁量に委ねる。上記のガードレールだけ守ること。

---

## 8. 期待する成果物

最低限、以下のデザイン（デスクトップ＋モバイル、ライト＋ダーク）:

1. **ワークスペース全体**（パレットバー＋メインのカードスクロール）
2. **パレットバーの各状態**: 単色／ペア（FG・BGバッジ）／パレット、空状態（色ゼロ）、色追加時
3. **モード選択UI**（2軸トグル）の表現
4. **代表的なカード**を数種:
   - 数値系（HSV／相対輝度のようなメーター付き）
   - 判定系（WCAGコントラスト比：比＋AA/AAAバッジ＋テキストプレビュー）
   - 可視化系（色相環 or 色差マトリクス）
5. **カラーピッカー**（追加・編集UI、HEX入力含む）
6. **確認ダイアログ**（破壊的操作時）
7. **指標ヘルプ**の表示形式（ツールチップ／ポップオーバー）

## 10. 用語ミニ辞書（デザインAI向け）

- **コントラスト比**: 2色の明暗差。WCAGで可読性基準（AA=4.5:1等）が定義される。1〜21の値。
- **HSL / HSV**: 色相(Hue)・彩度(Saturation)・輝度/明度 で色を表す数値表現。
- **相対輝度**: 色の知覚的な明るさ（0〜1）。コントラスト計算の基礎。
- **色差(ΔE)**: 2色の知覚的な違いの大きさ。小さいほど似ている。
- **色覚シミュレーション**: P型/D型/T型の色覚での見え方の再現。
- **調和スキーム**: 補色・類似色・トライアドなど、色相関係に基づく配色の型。
