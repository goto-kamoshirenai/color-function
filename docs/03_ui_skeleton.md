# 03. 基本設計 — UI骨格

> ステータス: ドラフト / 最終更新: 2026-06-10
> 関連: [コンセプト](./01_concept.md) / [制約・前提](./02_constraints.md)

コンセプト §6（単色→ペア→パレットの段階的統合）と §5（検証と設計の両輪）を、
具体的な画面の骨格に落とす。本書はレイアウト・状態・モードの「骨格」までを扱い、
個々のカードの中身（指標の計算仕様）は後続文書に委ねる。

---

## 1. レイアウト全体

画面は **常設の配色パレットバー（下部）** と **分析・編集領域（メイン）** の2層を基本とする。
パレットバーはほぼ全ページで常に表示され、グローバルな配色状態を保持・操作する。

```
┌──────────────────────────────────────────────┐
│  ヘッダー（任意・ページナビ／about・help 等）      │   ← 補助
├──────────────────────────────────────────────┤
│                                              │
│   メイン領域（分析・編集カードを縦スクロール）       │   ← 主役
│   現在のモードに該当するカードだけを並べる           │
│                                              │
│        ┌─コントラスト─┐                        │
│        │ 4.8:1  AA ✓ │                        │
│        └────────────┘                        │
│        ┌─HSV────────┐                        │
│        │ H210 S80 V60│        ↓ スクロール       │
│        └────────────┘                        │
│                                              │
├──────────────────────────────────────────────┤
│ パレットバー（常設・下部・コンパクト）              │   ← 常設・グローバル
│ [単位 単色|ペア|パレット] [観点 検証|設計] [＋]    │
│ ● ● ● ● ●   ← 色スウォッチ（選択・役割表示）       │
└──────────────────────────────────────────────┘
```

### 領域の責務

| 領域                       | 責務                                                       | 保持する状態                                         |
| -------------------------- | ---------------------------------------------------------- | ---------------------------------------------------- |
| パレットバー（下部・常設） | グローバル配色の保持・表示、色の選択、モード切替、色の追加 | `palette` / `selection` / `mode`                     |
| メイン領域                 | 現在のモードに対応する分析カード・編集UIをリアルタイム描画 | （グローバル状態を購読するだけ。ローカル状態は最小） |
| ヘッダー（任意）           | ページ間ナビ（about / help / contact 等）                  | ルーティングのみ                                     |

> メイン領域は**状態を所有しない**。常にグローバル状態を購読して描画する純粋な投影面にする。
> これにより「どのページ・どのカードからでも同じ色を見ている」一貫性が保たれる。

## 2. グローバル状態モデル

パレットバーが保持し、アプリ全体で共有する中核状態。

```ts
type Color = {
  id: string; // 安定した識別子（並べ替え・役割割当のキー）
  value: string; // 正準表現（例: HEX）。各指標はここから算出
  // 表示名や由来（色名辞書ヒット等）は派生情報として算出し、状態には持たない
};

type Unit = "単色" | "ペア" | "パレット";
type View = "検証" | "設計";

type AppState = {
  palette: Color[]; // グローバルな配色（順序つき・真実の源）
  selection: string[]; // フォーカス中の color.id 群（単位により意味が変わる §4）
  roles?: { fg?: string; bg?: string }; // ペア時の役割割当（前景/背景）
  mode: { unit: Unit; view: View }; // 2軸モード（§3）
};
```

- `palette` が **唯一の真実の源（single source of truth）**。
- 各種指標（コントラスト・色差・輝度…）は `palette` + `selection` から**算出するだけ**で、状態には持たない。
- 状態は [制約 §2.3](./02_constraints.md) のとおり **URL / localStorage** に同期し、DB を介さず共有・復元できる。
  - `palette` と `mode` は URL エンコード対象の最有力候補（コンパクトさを優先）。

## 3. モード ── 2軸（単位 × 観点）

モードは **「どの粒度で見るか（単位）」×「何の目的で見るか（観点）」** の2軸で構成する。
パレットバー内に2つのトグルとして設置する。

```
単位:  [ 単色 ] [ ペア ] [ パレット ]      ← 粒度
観点:  [ 検証 ] [ 設計 ]                  ← 目的
```

この2軸の組み合わせ（3 × 2 = 6）が、メイン領域に並ぶカード集合を決定する。

### モード × 表示カード マトリクス（初期方針）

|              | 検証（妥当性の診断）                                                              | 設計（配色を作る支援）                                      |
| ------------ | --------------------------------------------------------------------------------- | ----------------------------------------------------------- |
| **単色**     | 相対輝度 / HSL・HSV値 / 色名（辞書ヒット）                                        | トーン・明度彩度バリエーション展開 / 調和ルールから配色生成 |
| **ペア**     | WCAGコントラスト比（AA/AAA） / CIEDE2000色差 / 色覚での識別性                     | 補色・類似色などの相手色提案 / 前景・背景の役割入替         |
| **パレット** | コントラスト網羅マトリクス / 色差マトリクス / 色覚シミュレーション / エントロピー | 調和ルール適用 / 不足色の補完提案 / 配色テンプレート適用    |

> このマトリクスは**確定仕様ではなく初期配置案**。各カードの帰属モードは後続文書で精緻化する。
> 「検証と設計は地続き」（コンセプト §5.2）を保つため、観点トグルは**同じ単位・同じ色を保ったまま**
> 検証⇄設計を切り替えるだけにする（色や選択は失わない）。

## 4. 選択モデル（単位ごとの意味）

`selection` の意味は単位によって変わる。単位を切り替えても、可能な限り選択は引き継ぐ。

| 単位     | selection の意味                     | 操作                                              |
| -------- | ------------------------------------ | ------------------------------------------------- |
| 単色     | フォーカス1色                        | スウォッチをクリックで選択                        |
| ペア     | 前景・背景の2色（`roles`で役割保持） | 2つ選択 → 役割割当。入替ボタンで fg/bg をスワップ |
| パレット | 全色が対象（selection は強調用途）   | 個別ハイライトのみ                                |

- 単位を **単色→ペア** に上げると、フォーカス色を fg として保持し、bg を追加選択させる、等の引き継ぎ規則を定める（詳細は後続）。
- ペア→パレットでは選択はハイライトに退避し、評価対象は palette 全体になる。

## 5. カードシステム（拡張の骨格）

メイン領域は「カードの集合」であり、各分析軸＝1カードとして実装する。
各カードは **自分が表示される (単位, 観点) の組** を宣言し、メイン領域はモードでフィルタして並べる。

```ts
type Card = {
  key: string;
  appliesTo: { unit: Unit; view: View }[]; // 表示条件（複数モードに跨ってよい）
  // render: グローバル状態を購読し、指標を算出して描画
};
```

- 新しい可視化軸の追加 = **カードを1つ足して `appliesTo` を宣言するだけ**。レイアウトは触らない。
- 旧バージョンの `Card*`（コントラスト/HSV/CIEDE2000/相対輝度/エントロピー/プレビュー等）は、この枠に再配置する。
- カード内に編集UI（ピッカー等）を持ってよい（§6）。

## 6. 編集フロー ── どこからでもグローバル値を編集

色の編集は特定の場所に限定せず、**どこからでもグローバル値を編集できる**ことを基本とする。
スウォッチ・カード内ピッカー・数値入力など、いずれの入口からの変更も同じ中央アクションに集約する。

```
編集入口（スウォッチ / カード内ピッカー / 数値入力 …）
        │  変更意図 (intent)
        ▼
  applyColorChange(intent)        ← 中央アクション（唯一の更新経路）
        │
        ├─ 確認が必要なケース？ ──→ [確認ダイアログ] ──┐
        │                          （反映してよいか）   │
        ▼                                              ▼
   palette を更新（グローバル反映）  ◀───────────────  承認
        │
        ▼
   全カードがリアルタイムに再算出・再描画
```

- すべての編集は **`applyColorChange` 一本に集約**し、更新経路を一元化する（追跡・取消・URL同期がしやすい）。
- **確認ダイアログ**は「場合によって」差し込む（コンセプト議論での要望）。差し込む条件は後続で定義するが、候補は:
  - 既存配色を大きく破壊する一括変更（テンプレート適用・調和ルールの全置換など）
  - 復元しづらい操作（パレット全消去・並べ替えの確定 など）
  - 通常の1色微調整など影響が局所的な編集は**ダイアログなしで即時反映**（リアルタイム性を損なわない）

> 既定は「即時反映」。確認ダイアログは**例外的なガード**であり、常時介在させない。

## 7. ページ構成とバーの表示制御

「ほぼすべてのページ」での常設を満たすため、パレットバーは**アプリシェル（共通レイアウト）**に置く。

| ページ                   | パレットバー     | 備考                                         |
| ------------------------ | ---------------- | -------------------------------------------- |
| ワークスペース（主画面） | 表示             | メイン領域＝分析・編集カード                 |
| about / help             | 表示（既定）     | 解説中も配色を参照できる利点                 |
| contact 等               | 非表示にしてよい | ルート単位でバー表示をオプトアウト可能にする |

- 実装上はレイアウトでバーを描画し、**ルートごとに表示/非表示フラグ**を持てるようにする（既定: 表示）。

---

## 8. この骨格が後続設計に与える含意

- カード＝モードでフィルタされる宣言的部品 → **可視化軸の追加コストが低い**（コンセプト「考え得る限り可視化」と整合）。
- グローバル状態が `palette` 中心の薄い構造 → **URLエンコード・共有が容易**（制約 §2.3 と整合）。
- 更新が `applyColorChange` 一本 → **取消・履歴・確認ダイアログ・URL同期**を一箇所で扱える。

---

## 9. 未決事項（次に詰める）

1. **モード×カードの確定マトリクス**：各指標カードの帰属モードと、MVPに含めるカードの選定。
2. **選択引き継ぎ規則**：単位を上下に切り替えたときの selection / roles の具体的な遷移。
3. **確認ダイアログの発火条件**：どの操作を「破壊的」とみなすかの線引き。
4. **状態のエンコード仕様**：palette / mode を URL に載せる具体フォーマット（制約 §5 と連動）。
5. **パレットバーの具体UI**：スウォッチ数の上限、追加/削除/並べ替えの操作系、レスポンシブ時の畳み方。
