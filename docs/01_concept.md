# 01. コンセプト — Color Follows Function

> ステータス: ドラフト / 最終更新: 2026-06-10
> 関連: [制約・前提](./02_constraints.md)

---

## 1. ひとことで

**Color Follows Function** は、配色を「感覚」ではなく「数値」で語れるようにするための設計・検証ツールである。
デザインや Web 開発で発生する「この色で合っているのか」「なぜ読みづらいのか」という問いに対して、
コントラスト比・色差・色覚シミュレーションなどの定量指標で根拠を返す。

名前は **"Form Follows Function"（形態は機能に従う）** のもじりであり、
「色もまた、機能（伝えたいこと・読ませたいこと）に従って選ばれるべき」という思想を表す。

---

## 2. 背景と課題

色は、デザイン要素の中でもっとも感覚的・主観的に扱われやすい。

- 「なんとなく綺麗」「印象が良い」で配色が決まり、**根拠を言語化・数値化できない**。
- その結果、**アクセシビリティ（コントラスト不足・色覚多様性）の見落とし**が起きる。
- 配色の「良し悪し」を **チームで議論する共通言語がない**。レビューが好みのぶつけ合いになる。
- 既存ツールは「コントラスト比だけ」「カラーピッカーだけ」と**機能が分断**しており、
  単色の分析からパレット全体の評価までを一気通貫で見られる場が少ない。

本アプリは、これらを **「色の定量化・可視化」という一点**で束ねる。

---

## 3. コアコンセプト

> **感覚的に扱われがちな「色」を、考え得る限りの方法で数値的・定量的に可視化する。**

この一文が、機能採否の判断基準（ゲート）になる。
新機能を検討するとき、常に問う ──「これは色を**より定量的に理解・判断**できるようにするか？」。

- 単に綺麗なカラーピッカーを作ることが目的ではない。
- **数値・指標・可視化を通じて、配色の意思決定に根拠を与えること**が目的である。

---

## 4. 対象ユーザー

| ユーザー | 主なシーン | 求めるもの |
|---------|----------|----------|
| Web デザイナー | 配色設計・UI トーン決め | 調和した配色を効率よく作り、根拠を持ちたい |
| フロントエンド開発者 | 実装時の色の妥当性確認 | WCAG 準拠の確認、トークン化された色 |
| デザインレビュアー / ディレクター | レビュー・意思決定 | 「なぜこの色か」を数値で説明・合意したい |

共通項は **「色を主観でなく数値で扱いたい人」**。
特別な色彩工学の知識がなくても、指標の意味がその場で分かる（ヘルプ／解説を併設する）ことを重視する。

---

## 5. 提供価値 ── 検証と設計の両輪

本アプリは、以下の2つのモードを**同格**で提供する。両者は地続きであり、行き来できる。

### 5.1 検証（Validation）
既にある配色が妥当かを数値で診断する。
- WCAG コントラスト比による可読性判定（AA / AAA）
- 色覚多様性シミュレーション（P型・D型・T型）での見え方確認
- パレット内の色差（CIEDE2000）による「似すぎ／離れすぎ」の検出
- 相対輝度・エントロピーなどによる特性の可視化

### 5.2 設計（Design）
良い配色を作る支援をする。
- 調和ルール（補色・トライアド・類似色など）に基づく配色の展開
- 1色を起点としたトーン／明度・彩度バリエーションの生成
- 用途別の配色テンプレートからの出発

> 検証と設計は分断しない。**設計した配色はその場で検証され、検証結果を見て設計に戻れる**。
> これが「両輪で同格」の意味であり、本アプリの体験の核である。

## 6. 基本単位の考え方 ── 単色 → ペア → パレットの段階的統合

操作対象（基本単位）は、粒度の異なる3階層を**1つのワークスペース上で段階的に統合**して扱う。

```
 単色          ペア                  パレット
 ┌───┐        ┌───┬───┐            ┌───┬───┬───┬───┐
 │ ● │   →   │前 │背 │     →     │ ● │ ● │ ● │ ● │
 └───┘        └───┴───┘            └───┴───┴───┴───┘
 1色の         2色の関係            集合としての
 性質を分析    （主にコントラスト）  バランス・調和・網羅性
```

- **単色**: 1色の性質（HSL/HSV、相対輝度、色名など）を分析し、そこから色を展開する起点。
- **ペア**: 2色の関係を見る単位。中心はコントラスト比だが、色差や調和もここで効く。
- **パレット**: 色の集合としての全体評価。バランス・コントラスト網羅性・調和・色覚での識別性。

> ユーザーは単色から始めてペア・パレットへ育てることも、いきなりパレットを評価することもできる。
> 階層は**断絶した別画面ではなく、同じ作業空間の中で連続している**ことを前提に設計する。

## 7. 可視化の軸（候補カタログ）

「考え得る限り可視化する」という方針に沿った、可視化軸のカタログ。
（実装優先度は基本設計で決める。ここでは"何が可視化対象になりうるか"を網羅する。）

| 軸 | 何を見るか | 単位 |
|----|----------|------|
| WCAG コントラスト比 | 前景/背景の可読性（AA / AAA 判定） | ペア |
| 相対輝度 | 色の明るさ（知覚輝度の基礎値） | 単色 |
| HSL / HSV | 色相・彩度・明度／明るさの数値表現 | 単色 |
| 色相環（Hue Wheel） | 色相の位置関係・調和の可視化 | 単色〜パレット |
| CIEDE2000 色差 | 2色の知覚的な違いの大きさ | ペア〜パレット |
| 色覚多様性シミュレーション | P/D/T 型での見え方・識別性 | 全階層 |
| エントロピー | 配色の情報量・複雑さの指標 | パレット |
| トーン / バリエーション展開 | 明度・彩度を変えた派生色群 | 単色起点 |
| プレビュー（SVG/UI 模擬） | 実際の使用文脈での見え方 | ペア〜パレット |

> 旧バージョンで実装済みの軸（コントラスト・HSV・CIEDE2000・相対輝度・エントロピー・色の展開・色相環・プレビュー）は
> この方向性の実証であり、V3 では**これらを「単色→ペア→パレット」の連続体の中に再配置する**。

## 8. スコープ（コンセプトレベル）

### やること
- 色の定量分析と可視化（上記カタログ）
- 検証と設計を行き来できる単一ワークスペース
- 指標の意味をその場で学べる解説の併設
- 色名・基準値・調和ルール・色覚プロファイルを根拠データとして内蔵

### やらないこと（現時点）
- 画像からの自動配色抽出（将来検討の余地はあるが初期スコープ外）
- アカウント機能・サーバ側でのユーザーデータ保存（→ [制約・前提](./02_constraints.md)）
- 印刷向けカラーマネジメント（CMYK/ICC プロファイル等）の厳密対応
- 複数人リアルタイム共同編集

> 技術的な制約・前提（DB を持たない、状態の持ち方、エクスポート方針など）は
> コンセプトから切り離し、[02. 制約・前提](./02_constraints.md) に記載する。

---

## 9. 体験のゴール（成功の定義）

このアプリがうまくいっている状態とは：

1. ユーザーが「なんとなく」で選んでいた色に、**数値の根拠を添えて説明できる**ようになる。
2. アクセシビリティ上の問題（コントラスト不足・色覚での識別困難）を**作る前に気づける**。
3. 配色レビューが「好みの議論」から「**指標に基づく議論**」に変わる。
4. 検証して終わりではなく、**その場で直して再検証する**サイクルが自然に回る。

---

## 10. 未決事項 / 基本設計で詰めること

- 単色・ペア・パレットを「同じ作業空間で連続させる」具体的な画面構成（UI の骨格）。
- 可視化軸の**実装優先度**と、MVP に含める範囲。
- 状態の保存・共有方法（→ 制約文書と連動）。
- 解説（ヘルプ）の提供形態（モーダル／インライン／用語集）。
- 配色テンプレート・調和ルールの**具体的な定義粒度**。
